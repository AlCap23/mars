6 ## Introduction
7 
8 There are different possibilities how the MARS simulation can be
9 used. One possibility is to write a plugin that handles the loading of
10 a scenario and interfaces the simulation to some control software.
11 Also, in many cases, the control software can be directly wrapped into
12 a simulation plugin. This tutorial will guide you how to create a new
13 plugin, load your scene, and control the simulation.
14 
15 ## Prerequisites
16 
17 We assume you already have a MARS development environment setup in
18 "$MARS_DEV_ROOT". You should also have a simulation scene or SMURF file
19 to load into the simulation.
20 Open a terminal and source your environment:
21 
22  cd $MARS_DEV_ROOT;
23  . env.sh
24 
25 ## Basic Plugin
26 
27 ### Create and build a new plugin
28 
29 To setup your first plugin you can use a script wich will create a new
30 plugin:
31 
32  cd simulation/mars/plugins/plugin_template
33  ./cnp.sh
34 
35 The script will ask you for the project name, from which it will also
36 try to derive the class name of the main c++ class.
37 
38 > NOTE: In project names, be sure to avoid special
39 characters such as **"**, **&** and so on. They may cause problems in the
40 subsequently executed scripts or the file system, as the name entered is
41 used to create variables and file/folder names.
42 Dashes ( **-** ) and underscores ( **_** ) are fine.
43 
44  E.g. we can enter
45 "basic_plugin" here and the main class will then be called
46 "BasicPlugin". Then you can enter a description of your plugin ("The
47 plugin created by following the guide of the basic plugin tutorial.").
48 Afterwards, enter the author name, email, and confirm your data. The
49 script will create a folder containing your plugin one directory above
50 the script location ("$MARS_DEV_ROOT/mars/plugins/basic_plugin"). You
51 can go into that folder and use the build.sh script to build the
52 plugin for a first test if everything went well. To install the
53 plugin you have to use "make install" within the newly created build
54 folder.
55 
56  cd $MARS_DEV_ROOT/mars/plugins/basic_plugin
57  ./build.sh
58  cd build
59  make install
60 
61 If you want to create a custom configuration to work with your plugin,
62 copy the "mars_default" configuration to "mars_basic_plugin", rename your
63 example additional lib configuration file "other_libs.txt.example" into "other_libs.txt" and add your plugin to the list in the file. This will
64 cause your plugin to be loaded upon startup of MARS.
65 
66  cd $MARS_DEV_ROOT/install/configuration
67  cp -r mars_default mars_basic_plugin
68  cd mars_basic_plugin
69 
70 Content of other_libs.txt:
71 
72  ...
73  connexion_plugin
74  data_broker_gui
75  cfg_manager_gui
76  lib_manager_gui
77  basic_plugin
78 
79 Start the simulation:
80 
81  mars_app
82 
83 Currently the plugin does not do anything, but it should show up in
84 the list of loaded plugins which you can find under the menu entry:
85 File->Library Info...

88 ### Adapt the plugin
89 
90 In the "$MARS_DEV_ROOT/mars/plugins/basic_plugin/src/BasicPlugin.cpp"
91 you will find a "init()" function that is called by the simulation to
92 initialize the plugin. If your initialization code depends on the
93 simulation or other plugins you should rather perform it in the init()
94 function than in the constructor as the other modules may not be in a
95 defined state, yet. Just comment in the loadScene line with the path to
96 your scene file.
97 
98 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
99 void BasicPlugin::init() {
100  // Load a scene file:
101  control->sim->loadScene("robo.scn");
102 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
103 
104 In this tutorial we are going to access the motors and sensors of the
105 simulation. To do that we first need to include the corresponding
106 interfaces:
107 
108 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
109 #include <mars/interfaces/sim/MotorManagerInterface.h>
110 #include <mars/interfaces/sim/SensorManagerInterface.h>
111 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
112 
113 To control the robot in the simulation you can set motor values in the
114 update callback. The update callback is triggered by the simulation
115 thread giving the simulation step time as parameter. Here we can set
116 some motor values:
117 
118 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
119 void BasicPlugin::update(sReal time_ms) {
120 
121  control->motors->setMotorValue(1, 1.0);
122  control->motors->setMotorValue(2, 3.0);
123 }
124 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
125 
126 After doing "make install" in
127 "$MARS_DEV_ROOT/mars/plugins/basic_plugin/build" we should have a
128 small robot driving in circles when we start the simulation.
129 
130 In the next step we want to read the sensor values from the laser
131 scanner to create a wall following behavior. We can access the sensor
132 values by the id the sensor gets when it is loaded into the
133 simulation:
134 
135 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
136 void BasicPlugin::update(sReal time_ms) {
137  static unsigned long laserId = control->sensors->getSensorID("laser");
138 
139  control->motors->setMotorValue(1, 1.0);
140  control->motors->setMotorValue(2, 3.0);
141 }
142 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
143 
144 The "getSensorData()" methods takes the sensor id and a pointer to a
145 unallocated sReal pointer as arguments. It will allocate memory for
146 the sensor data but the caller (in this case that is you) is
147 responsible for freeing the memory after reading the sensor data. The
148 method returns the number of values representing the sensor data.
149 
150 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~{.cpp}
151 void BasicPlugin::update(sReal time_ms) {
152  static unsigned long laserId = control->sensors->getSensorID("laser");
153  sReal *sensorData;
154  int numSensorValues = control->sensors->getSensorData(laserId, &sensorData);
155  assert(numSensorValues == 8);
156  if(sensorData[3] < 1.0 || sensorData[0] < 0.4) {
157  control->motors->setMotorValue(2, 12.0);
158  } else if(sensorData[0] > 0.7) {
159  control->motors->setMotorValue(2, 4.8);
160  } else {
161  control->motors->setMotorValue(2, 5.0);
162  }
163  control->motors->setMotorValue(1, 5.0);
164  free(sensorData);
165 }
166 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
167 
168 Now we can do "make install" again, start the simulation, and we
169 should see the robot following the wall!
